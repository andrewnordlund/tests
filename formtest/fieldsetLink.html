Fieldset with Link

This page is to test how various AT deals with links being in <fieldset>/<legend>s; having an ordered list as part of a <legend> and referring to parts of the <legend> using aria-describedby in the radio buttons' <label>.

Test Code


This form doesn't need a submit button.

Radio buttons. Ahh yes some radio buttons, and there ain't no link here.












This form doesn't need a submit button.

Radio buttons. Ahh yes Loveley, wonderful Radio buttons.











Please put the following Loveley, wonderful Radio buttons in order. Item 1 Item 2 Item 3







Results

VoiceOver on iOS 14.7

  • VoiceOver (VO) only reads the <legend> text when swipped-right/left to; not along with radio button <label>s. The <legend> text is also skipped when the rotator has selected "Form controls" and up/down swipping is used.
  • When there's a link in the <legend>, left/right-swipping reads 1 piece at a time. So it reads the <legend> text before the link, then swipe-right, then the link text, then "link", then swipe left, then the <legend> text that comes after the link
  • For the role="list, VoiceOver treats that as an actual list, reading out the number, the period after the number, then the list item content aling with "list start" and "list end" at the first and last part of the list.
  • For the aria-describedby parts in the list items, it reads out "3 item 3 then 1 item 1 then 2 item 2..."
  • There is no difference in behavior in Safari, Firefox, or Chrome on iOS 14.7

VoiceOver on OS X Catalina

  • VoiceOver (VO) reads the <legend> only when tabbing into a <fieldset> from above or below. And it reads it last. On Safari, the <legend> is read out twice.
  • When Tabbing from the "Name 1" <input> to the link in the <legend>
    Firefox and Chrome:
    first VO reads "Link" then the link text then the whole <legend> text then "group"
    In Safari
    VO does the same as Firefox and Chrome, but reads out the the <legend> text twice before saying "group".
  • Then when you Tab into the radio buttons:
    Firefox and Chrome:
    VO reads out the <label> text, "radio button, x of y".
    Safari:
    VO reads the same as Firefox and Chrome, then follows that up with the <legend> text.
  • When Shift+Tabing from the radio buttons to the <legend>'s link:
    Firefox and Chrome:
    VO reads out "link" followed by the link text.
    Safari:
    VO does the same as Firefox and Chrome, and then reads out the <legend> text.
    VO does not read out that you're in a <legend>.
  • Then when Shift+Tabbing to the radio buttons from below:
    Firefox and Chrome:
    VO reads out the <label> text, "radio button, x of y", then the <legend> text followed by "group".
    Safari:
    The same as Firefox and Chrome, but reads out the <legend> text twice before reading out "group".
    There's no mention of a link in the <legend>.
  • When there's a link in the legend and there's an role="list" in the <legend>, when the link is Tab'd to:
    Firefox:
    the <legend> is read out as above ("link" + link text + <legend> text + "group". The list items are read out with their number (ie: "1. item 1 2. item 2 3. item 3" - and there's a pause at each period).
    Chrome:
    the <legend> is read out as above ("link" + link text + <legend> text + "group". The list items are read out without their number (ie: "item 1 item 2 item 3")
    Safari:
    VO reads out "link" then the link text then the <legend> text (twice) with the word "group". Sarari does not read out the list! In order to hear that list at this point, you have to use the ctrl+option+arrow keys instead of Tab. (But you will hear the list when you Tab to the radio buttons.
  • When Tabing to the radio buttons that have aria-describedby:
    Firefox:
    first the <label> text is read followed by "radio button x of y", then it reads out all the aria-describedby contents, including the display:none "then" <span>. ie: "1, 3, 2 radio butto 1 of 4 1 item 1 then 3 item 3 then 2 item two".
    Chrome:
    first the <label> text is read followed by "radio button x of y", then it reads out all the aria-describedby contents, including the display:none "then" <span>. But it only reads out the "then" once. ie: "1, 3, 2 radio butto 1 of 4 1 item 1 then 3 item 3 2 item two".
    Safari:
    first the <label> text is read followed by "radio button x of y", then it reads the <legend> text without the list. But then it reads out all the aria-describedby contents, including the display:none "then" <span>. ie: "1, 3, 2 radio butto 1 of 4. Please put the following Loveley, wonderful Radio buttons in order. 1 item 1 then 3 item 3 then 2 item two".
    No mention of the link in the <legend>.
  • When Shift+Tabing into the <fieldset> from below:
    Firefox:
    first the <label> text is read followed by "radio button x of y" followed by the <legend> text then "group". Then it reads out all the aria-describedby contents including the display:none "then" <span>. ie: "3, 1, 2 radio button 4 of 4, Please put the following Lovely, wonderful Radio buttons in order. 1. item 1 2. item 2 3. item 3 group. 3. item 3 then 1. item 1 then 2. item 2"
    Chrome:
    first the <label> text is read followed by "radio button x of y" followed by the <legend> text then "group". Then it reads out all the aria-describedby contents including the display:none "then" <span> - but only once. ie: "3, 1, 2 radio button 4 of 4, Please put the following Lovely, wonderful Radio buttons in order. 1. item 1 2. item 2 3. item 3 group. 3. item 3 then 1. item 1 2. item 2"
    Safari:
    first the <label> text is read followed by "radio button x of y" followed by the <legend> text twice then "group". ie: "3, 1, 2 radio button 4 of 4, Please put the following Lovely, wonderful Radio buttons in order. Please put the following Lovely, wonderful Radio buttons in order. group. 3. item 3 then 1. item 1 then 2. item 2"
    No mention of the link is made.

NVDA 2021.1 on Windows 10 (IE 11, Firefox 90.0.2 and 55.0.3, Chrome 92, Edge 92)

  • NVDA reads the <legend> only when tabbing into a <fieldset> from above or below. And it reads it first.
  • When there's a link in the <legend>, and you Tab to the link:
    Firefox and Chromium:
    first it reads out the <legend> text with the word "grouping", then link text followed by the word "link".
    IE:
    NVDA just reads the link text with "link". It doesn't read out the <legend> text.
  • Then when Tabbing to the first radio button:
    Firefox, Chomium and IE:
    the <legend> text is not read in any browser.
  • If you Shift+TAb from the radio buttons to the link in the <legend>:
    Firefox, Chromium, and IE:
    the link is read out as a normal link (link text + "link").
  • When Shift+Tabbing to the radio buttons from below:
    Firefox, Chromium, and IE:
    the <legend> text is read with "grouping" followed by the selected radio button <label> text and its status.
    No mention of the link in the <legend>.
  • When there's a link in the <legend> and there's an role="list" in the <legend>, when the link is Tab'd to:
    Firefox, and Chromium:
    the <legend> text including the aria list as an ordered list + the link text + "link". The list items are read out with their number (ie: "1 item 1").
    IE:
    NVDA only reads out the link text + "link". The list is ignored.
  • When Tabing to the radio buttons that have aria-describedby:
    Firefox:
    first the <label> text is read followed by "radio button (not) checked". Then it reads out all the aria-describedby contents, including the display:none "then" <span> followed by "x of y".
    Chromium:
    first the <label> text is read followed by "radio button (not) checked". Then it reads out all the aria-describedby contents, including the display:none "then" <span> - but only the first time - followed by "x of y". Ex: "1. item 1 then 2. item 2 3. item 3".
    IE:
    NVDA does not read out the list item number, nor does it read out "then", nor the "x of y". ex: "1, 3, 2 radio button not checked item 1 item 3 item 2".
  • When Shift+Tabing into the <fieldset> from below:
    Firefox:
    First the <legend> text is read followed by "grouping" followed by the <label> text + "radio button (not) checked". Then it reads out all the aria-describedby contents as above with all the "then"s followed by "x of y". ie: "Please put the following Lovely, wonderful Radio buttons in order. 1. Item 1. 2. Item 2. 3. Item 3 grouping. 1, 3, 2 radio button not checked 1. item 1 then 3. item 3 then 2. item 2."
    Chromium:
    First the <legend> text is read followed by "grouping" followed by the <label> text + "radio button (not) checked". Then it reads out all the aria-describedby contents as above with the "then" the first time followed by "x of y". ie: "Please put the following Lovely, wonderful Radio buttons in order. 1. Item 1. 2. Item 2. 3. Item 3 grouping. 1, 3, 2 radio button not checked 1. item 1 then 3. item 3 2. item 2."
    IE:
    First the <legend> text is read followed by "grouping" followed by the <label> text + "radio button (not) checked". Then it reads out all the aria-describedby contents as above without all the "then"s. ie: "Please put the following Lovely, wonderful Radio buttons in order. Item 1. Item 2. Item 3 grouping. 1, 3, 2 radio button not checked item 1 item 3 item 2".

JAWS 2020.2004.66 on Windows 10

  • JAWS reads the <legend> only when tabbing into a <fieldset> from above or below. And it reads it first.
  • When Tabbing from the "Name 1" <input> to the link in the <legend>, things get interesting:
  • In IE, JAWS just reads out "Name 1 colon edit" - when the link is selected!
  • In Firefox JAWS reads out the <legend> text followed by the link text + "link". This is what we want.
  • In Chromium browsers (Chrome and Edge), JAWS reads out the <legend> text + "Name 1 colon edit".
  • When Tabbing from the link to the radio buttons:
    In IE:
    JAWS now reads out the <legend> text followed by the selected radio button's <label> text + " radio button (not) checked". There's no mention of a link.
  • In Firefox, JAWS reads out the <label> of the first radio button followed by the <label> of the selected radio button. So if "Option 3" is selected, JAWS reads out "Option 1 not checked Option 3 radio button checked 3 of 6."
  • In Chromium browsers, it just reads out "Option x radio button checked, x of y"
  • When there's a link in the <legend>, and you Shift+Tab to the link from a radio button:
    In IE:
    JAWS reads out the <legend> text plus the <label> of the checked radio button and its "checked" status, ex: "Radio buttons, Ahh yes Lovely, wonderful Radio buttons. Option 4 radio button checked." No mention of a link.
    In Firefox:
    it reads the link text with "link". None of the rest of the <legend> text is read. It's not indicated that you're even in a <legend> or <fieldset>.
    In Chromium it reads the <label> of the selected radio button with x of y. ex:
    ""Option 6 radio button checked 6 of 6." If no radio button is selected, instead it reads out the last focused radio button's <label> with "not checked" + "x of y". No mention of a link or <legend> or link.
  • Then when Shift+Tabbing to the radio buttons from below:
    In IE:
    the <legend> text then the selected option is read with its status. No mention of a link.
    In Firefox:
    JAWS reads the <legend> text then the first radio button's <label> text followed by the selected radio button's <label> textwith their status. Ex: "Radio buttons. Ahh yes Lovely, wonderful Radio buttons. Option 1 radio button not checked. Option 3 radio button checked." No mention of a link.
    In Chromium:
    JAWS reads the <legend> text + the selected radio button's <label> text, its status, then "x of y". No mention of a link.
  • When there's a link in the <legend> and there's an role="list" in the <legend>, when the link is Tab'd to:
    In IE:
    JAWS just reads out the <label> of the previous <input>, as above. Note that if that <input> is removed, then when the focus lands on the link, the <legend> text of the previous <fieldset> is read along with the selected radio button. It's as though JAWS is just reading out the last <legend> or <label>. Incidentally, if I Alt+Tab away and back again, instead of the last <label> or <legend>, I get the browser name and page name.
    In Firefox:
    The <legend> text is read followed by the link text and the word "link". Note that JAWS also reads out the aria list as though it were a real <ol>. This is what we want.
    In Chromium:
    the <legend> text followed by "Name 2 colon edit.". JAWS does not read out the list item number like it does with Firefox.
  • When Tabing to the radio buttons that have aria-describedby:
    In IE:
    the <legend> text is read then the radio button's <label> + "radio button (not) checked". No mention of link. aria-describedby is ignored.
    In Firerox:
    JAWS reads out the <legend> text with the link text and "link" then the focused radio button's <label> + "radio button (not) checked" then it reads out the aria-describedby texts along with all the "then"s of the first radio button, followed by the same for the selected radio button.
    In Chromium:
    JAWS reads out the <label> of the focused radio button, along with the aria-describedby parts. The <legend> text is left out. Something weird happens here. If you use the up/down arrow keys to select another radio button, you hear "Blank" betwixt each radio button selection. And if you're sighted, you don't actually see your visual focus indicator moving around. Then if you hit (Shift+)Tab your focus goes to the radio button that JAWS has focused, and not to the link in the <legend>. But, the now focused radio button doesn't change it's checked status.
    Note that Chromium browsers (Chrome, Edge) only read out "then" the first time. ex:
    "1. item 1 then 2. item 2 3. item 3".
    Note that IE does not read out the list item number, nor does it read out "then", nor the "x of y". ex:
    "1, 3, 2 radio button not checked item 1 item 3 item 2".
  • When Shift+Tabing into the <fieldset> from below:
    Firefox
    JAWS reads out the <legend> text (no mention of link) along with the list item count, followed by the <label> of the first radio button and its status then the <label> of the selected radio button with its status and place in the list. Then it reads the aria-describedby text with all the "then"s.
    • Chromium:
    JAWS reads out the <legend> text (no mention of a link) along with the list item count, followed by the selected radio button's <label> followed by "radio button (not) checked x of y" then the aria-describedby text. But it only reads the "then" once. ex: "1. item 1 then 2 item 2 3 item 3".
    IE:
    the <legend> text is read (no mention of a link) followed by the selected radio button's <label> + "radio button checked" followed by the aria-describedby text - including all the "then"s. It does not read out the list item number, nor the "x of y". But it does it read out "then". ex:
    "Please put the following Lovely, wonderful Radio buttons in order. Item 1. Item 2. Item 3. 1, 3, 2 radio button not checked item 1 item 3 item 2". (No "grouping") ???
